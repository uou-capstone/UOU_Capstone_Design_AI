# 아이디어 실제 구현

이 스킬은 이미 정해진 아이디어를 "좋은 생각" 단계에서 끝내지 않고, 실제 시스템 변경까지 밀어붙일 때 사용한다.
핵심 목적은 아래 6단계를 하나의 엔지니어링 루프로 묶는 것이다.

1. 구현 가능성 판단과 계획 수립
2. 상세 design 문서 작성
3. 정적 검증 기반 설계 개선
4. 실제 구현
5. 구현 결과 재검토
6. 최종 design 문서와 최종 보고서 작성

## 언제 사용할지

- 브레인스토밍으로 나온 아이디어 중 하나를 실제 코드까지 연결해야 할 때
- 구현 전에 design 문서를 충분히 구체화해야 할 때
- 설계 단계와 구현 단계 모두에서 다수 관점의 리뷰가 필요할 때
- 최종적으로 개발자가 이해하기 쉬운 보고서와 design md를 남겨야 할 때
- "어떤 아이디어를 왜 이렇게 코드에 녹였는지"를 추적 가능한 형태로 정리해야 할 때

## 시작 전 입력 확인

작업 시작 전 아래를 먼저 확인한다.

- 구현할 대상 아이디어 또는 요구사항
- 읽어야 할 코드 범위와 관련 문서
- 저장할 최종 design 문서 경로
- 저장할 최종 보고서 경로
- sub-agent 사용 가능 여부

사용자가 경로를 명시하지 않았다면 한 번에 짧게 확인하거나, 바로 진행해야 하는 상황이면 아래 기본값을 사용한다.

- design 문서: `<output-dir>/implementation-design.md`
- 최종 보고서: `<output-dir>/implementation-report.md`

`<output-dir>`도 없으면 현재 작업 디렉토리 안의 관련 문서 디렉토리나 `capstone-study-docs` 계열 경로를 우선 사용한다.

사용자가 sub-agent 사용을 명시적으로 허용했다면 실제 sub-agent를 사용한다.
그렇지 않으면 같은 역할 분리를 유지한 채 순차적으로 시뮬레이션한다.

## 권장 preset

사용자가 세부 설정을 따로 주지 않았다면 아래 preset을 권장한다.

- 계획 단계: `3 agents x 3 rounds`
- design 정적 검증 단계: `10 agents x 2~3 loops`
- 구현 후 검토 단계: `10 agents x 2~3 loops`

질문이 필요하다면 가능하면 한 번에 끝낸다.

권장 질문 형식:

```text
구현 스킬을 시작하겠습니다.
구현할 아이디어와 읽어야 할 코드 범위를 확인하고,
최종 design 문서와 최종 보고서를 어디에 저장할지 정하면 바로 진행할 수 있습니다.
sub-agent는 계획 3x3, 검증 10명, 구현 후 검토 10명 기준으로 진행할까요?
```

## 전체 원칙

- 아이디어를 바로 구현하지 말고 먼저 현재 시스템과의 결합 방식을 이해한다.
- design 문서는 구현의 부산물이 아니라 구현을 이끄는 기준 문서여야 한다.
- 검증 단계에서 큰 문제가 나오면 억지로 구현을 이어가지 말고 design 단계로 되돌아간다.
- 구현 중 design과 현실 코드가 어긋나면 design 문서를 먼저 갱신한 뒤 구현을 정렬한다.
- 최종 산출물은 "무엇을 바꿨는지"뿐 아니라 "왜 그렇게 바꿨는지", "적용 후 시나리오가 어떻게 달라지는지"까지 설명해야 한다.
- 내부 메모나 임시 reasoning 로그는 기본적으로 영구 저장하지 않는다.

## 단계별 워크플로

### 0. Baseline scan

먼저 아래를 읽고 현재 상태를 잡는다.

- 관련 architecture/design 문서
- 구현 대상과 직접 연결된 코드
- 기존 brainstorm 결과나 아이디어 문서
- 필요하면 테스트, 실행 스크립트, 관찰 가능한 오류 로그

이 단계의 목표는 아래 4가지를 정리하는 것이다.

- 현재 시스템에서 어떤 모듈이 바뀌는가
- 아이디어의 핵심 가치가 어디에 들어가야 하는가
- 기존 구조와 충돌할 가능성이 있는 부분은 어디인가
- 구현 전에 먼저 풀어야 할 불확실성은 무엇인가

### 1. 구현 계획 수립

이 단계에서는 설계에 들어가기 전에 "어떻게 구현할지"를 먼저 계획한다.

기본 동작은 `3 agents x 3 rounds`다.

#### Planning round 1

각 planning agent는 독립적으로 아래를 작성한다.

- 문제 재정의
- 구현 대상 모듈 후보
- 변경 범위
- 위험 요소
- 가장 작은 구현 단위

#### Planning round 2

각 agent는 다른 agent 계획의 약점을 비판적으로 읽고 아래를 추가한다.

- 빠진 요구사항
- 충돌하는 설계 가정
- 과도한 복잡도
- 더 단순한 대안

#### Planning round 3

각 agent는 서로의 의견을 반영해 현실적인 통합안을 낸다.
메인 에이전트는 이를 병합해 최종 implementation plan을 정리한다.

최종 계획에는 최소한 아래가 있어야 한다.

- 목표와 비목표
- 변경 대상 모듈
- 의존성
- 단계별 구현 순서
- 예상 리스크
- 검증 전략

### 2. 상세 design 문서 작성

계획이 정리되면 메인 에이전트가 design 문서를 쓴다.
이 단계는 굳이 sub-agent 중심일 필요는 없고, 한 사람이 책임지고 일관된 문서를 만드는 편이 좋다.

design 문서에는 아래 항목을 권장한다.

- 문제 정의
- 목표 / 비목표
- 현재 구조 요약
- 제안 구조
- 데이터 흐름 또는 상태 흐름
- 핵심 인터페이스 변경점
- 모듈별 책임 변화
- 실패 시나리오와 복구 방식
- 테스트 전략
- 롤아웃 또는 마이그레이션 고려사항
- 사용자 시나리오 before / after

### 3. 정적 검증 루프

design 문서를 쓴 뒤 바로 구현하지 않는다.
먼저 다수 관점의 정적 검증으로 설계를 다듬는다.

권장 설정은 `10 agents`이며, 최소 `7 agents`까지 허용한다.

각 검증 agent는 서로 독립적으로 design 문서를 읽고 아래 lens 중 하나를 맡는다.

1. `Requirement coverage`
2. `Architecture consistency`
3. `State/Data correctness`
4. `Failure mode / recovery`
5. `Testability / observability`
6. `Performance / cost`
7. `Security / safety`
8. `Implementation feasibility`
9. `Developer ergonomics / maintainability`
10. `Product scenario fit`

검증 단계에서는 아래 규칙을 따른다.

1. 각 agent는 독립적으로 문제점을 찾는다.
2. 이후 서로의 지적을 비교해 중복과 우선순위를 정리한다.
3. 치명적이거나 구조적인 문제가 있으면 구현으로 넘어가지 않는다.
4. 메인 에이전트가 design 문서를 수정한다.
5. 큰 수정이 있었다면 다시 정적 검증을 반복한다.

정적 검증 종료 기준:

- 치명적 설계 모순이 없다.
- high severity 이슈가 해소되었거나 명시적으로 수용되었다.
- 구현 가능한 수준으로 인터페이스와 책임 분리가 명확하다.

### 4. 구현 단계

정적 검증이 수렴하면 design 문서를 기준으로 실제 구현을 시작한다.

구현 중에는 아래 원칙을 따른다.

- 큰 변경은 작은 단위로 나눈다.
- design 문서와 달라진 판단이 생기면 code만 바꾸지 말고 design도 함께 갱신한다.
- 테스트나 검증 가능한 경계부터 먼저 만든다.
- 새로 추가한 동작이 사용자 시나리오에서 어떻게 보이는지 계속 확인한다.

필요하면 구현 도중 아래를 갱신한다.

- 인터페이스 정의
- 상태 전이 설명
- 예외 처리 규칙
- 테스트 전략

### 5. 구현 후 재검토 루프

구현이 끝났다고 바로 종료하지 않는다.
다시 다수 agent 관점으로 "design vs code vs original idea"를 대조 검토한다.

권장 설정은 `10 agents`이며, 설계 정적 검증 단계와 비슷하게 운영한다.

각 review agent는 최소한 아래 중 하나를 맡는다.

1. `Design traceability`
2. `Behavior correctness`
3. `Regression risk`
4. `Edge case coverage`
5. `Test completeness`
6. `Documentation drift`
7. `Performance impact`
8. `Safety / guardrail`
9. `Operational readiness`
10. `Developer readability`

검토 결과 아래 문제가 나오면 구현 단계로 되돌아간다.

- design 문서와 실제 코드가 다르게 동작한다.
- 요구사항 일부가 빠졌다.
- 새로운 버그나 회귀 가능성이 생겼다.
- 설명 문서가 코드 현실을 따라가지 못한다.

### 6. 최종 산출물 작성

루프가 끝나면 아래 두 가지를 영구 저장한다.

1. 최종 design 문서
2. 최종 구현 보고서

중간 계획 메모, 임시 검증 메모, 라운드별 reasoning은 사용자가 요청하지 않는 한 저장하지 않는다.

## 최종 design 문서 규칙

최종 design 문서는 "구현 전에 상상했던 문서"가 아니라, 최종 코드 현실까지 반영된 기준 문서여야 한다.

권장 구조:

```md
# Implementation Design

## Summary
## Problem
## Goals / Non-goals
## Current System Snapshot
## Proposed Design
## Module Changes
## Data and Control Flow
## Failure Handling
## Test Strategy
## Scenario Changes
## Open Risks
```

## 최종 보고서 규칙

최종 보고서는 개발자와 협업자가 빠르게 읽고 이해할 수 있어야 한다.
아래 항목을 권장한다.

```md
# Implementation Report

## 한눈에 보기
- 무엇을 구현했는가
- 왜 이 방식으로 반영했는가

## Code Update Summary
- 바뀐 모듈
- 핵심 변경점

## Idea to Code Mapping
- 아이디어의 어떤 부분이 어떤 코드로 반영되었는가

## Scenario Before / After
- 적용 전 시스템 시나리오
- 적용 후 시스템 시나리오

## Developer Guide
- 코드의 어느 부분을 보면 전체 흐름을 이해할 수 있는가
- 핵심 엔트리 포인트와 책임 분리

## Verification Summary
- 어떤 검증을 했는가
- 남아 있는 리스크는 무엇인가
```

## 내부 메모 형식 예시

planning 또는 review 메모가 잠깐 필요할 때는 아래 정도의 간단한 형식을 사용한다.

```md
# Phase
## Lens
## Findings
## Risks
## Recommended change
## Go / No-go
```

이 메모는 내부 작업용이며, 기본적으로 최종 영구 문서에 남기지 않는다.

## 산출물 기본 규칙

- 최종 산출물은 사용자가 저장하라는 경로와 파일명에 맞춘다.
- 사용자가 최종 보고서만 파일명으로 지정했다면, design 문서는 같은 디렉토리에 짝이 되는 이름으로 저장한다.
- 사용자가 design 문서 경로만 지정했다면, 최종 보고서는 같은 디렉토리에 짝이 되는 이름으로 저장한다.
- 경로가 명확하지 않으면 먼저 짧게 확인하되, 반복 질문은 피한다.

## 이 스킬의 핵심 성공 기준

- 아이디어가 설계와 코드에 모두 traceable 하다.
- design 문서가 실제 코드와 어긋나지 않는다.
- 정적 검증과 구현 후 검토를 거쳐 치명적 허점이 줄어들었다.
- 최종 보고서를 읽으면 변경 이유와 시스템 시나리오 변화를 빠르게 이해할 수 있다.
