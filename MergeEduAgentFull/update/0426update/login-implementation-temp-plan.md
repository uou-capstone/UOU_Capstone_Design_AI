# 로그인 구현 임시 계획

## 범위

- 이메일 회원가입과 이메일 인증 코드 검증을 구현한다.
- 로그인/로그아웃/현재 사용자 조회를 구현한다.
- 회원가입 시 `teacher`와 `student` 역할을 선택하게 한다.
- 선생님은 강의실/주차/자료를 만들고 삭제할 수 있다.
- 학생은 초대받은 강의실만 보고, 자료 조회와 학습 세션만 사용할 수 있다.
- 학생 초대는 `이름 + 4자리 코드`로 검색한 뒤 강의실 등록 관계를 저장한다.
- Google 계정 가입/로그인은 Google OAuth 환경 변수가 설정되면 동작하는 서버 경로와 UI 진입점을 만든다.

## 비범위

- 실제 SMTP 메일 발송 연동은 이번 구현에서 제외한다.
- 운영용 관리자 콘솔과 역할 변경 승인 시스템은 제외한다.
- 기존 AI 에이전트의 설명/퀴즈 로직 자체는 변경하지 않는다.

## 인증 전략

- 서버가 발급한 불투명 세션 토큰을 HttpOnly 쿠키로 저장한다.
- 토큰 원문은 저장하지 않고 SHA-256 해시만 JSON 저장소에 저장한다.
- 이메일 비밀번호는 Node `crypto.scrypt`로 솔트 해시한다.
- 이메일 인증 코드는 해시해서 저장하고 만료 시간을 둔다.
- 개발 환경에서는 테스트 편의를 위해 회원가입 응답에 인증 코드를 반환한다.
- Google OAuth는 `GOOGLE_OAUTH_CLIENT_ID`, `GOOGLE_OAUTH_CLIENT_SECRET`, `GOOGLE_OAUTH_REDIRECT_URI`가 있을 때만 활성화한다.

## 데이터 변경

- `users.json` 추가
- `auth-sessions.json` 추가
- `classroom-enrollments.json` 추가
- `oauth-states.json` 추가
- `classrooms.json` 항목에 `teacherId` 추가
- 세션 상태에 `ownerUserId` 추가

## 서버 변경

- `AuthService` 추가
- `auth` 라우터 추가
- `requireAuth`, `requireTeacher`, 권한 검사 유틸 추가
- 강의실/주차/자료/세션 라우터에 역할별 접근 제어 추가
- `/uploads` 정적 공개 제공을 보호 라우트로 변경
- 학생 초대 API 추가

## 프론트엔드 변경

- `AuthProvider` 또는 인증 컨텍스트 추가
- 로그인/회원가입 화면 추가
- 보호 라우트 추가
- API 요청에서 세션 쿠키를 사용하도록 유지
- 선생님/학생 역할에 따라 대시보드와 강의실 UI 액션을 분기
- 선생님 강의실에 학생 초대 패널 추가
- 학생 계정에는 업로드/삭제/생성 UI를 숨김

## 검증 전략

- 서버 단위 테스트
  - 회원가입, 이메일 인증, 로그인, 로그아웃
  - 잘못된 비밀번호와 미인증 이메일 거부
  - 선생님/학생 역할별 강의실 목록
  - 학생 초대와 중복 초대 방지
  - 학생의 생성/삭제/업로드 API 거부
  - 사용자별 학습 세션 분리
- 웹 빌드
  - `npm run build`
- 브라우저 수동 테스트
  - 선생님 가입/인증/로그인
  - 강의실 생성
  - 학생 가입/인증/로그인
  - 학생이 초대 전에는 강의실을 볼 수 없는지 확인
  - 선생님이 학생을 초대
  - 학생이 초대 후 강의실과 자료를 볼 수 있는지 확인
  - 학생에게 생성/삭제/업로드 버튼이 없는지 확인

## 예상 위험

- 기존 데이터에는 `teacherId`가 없으므로 첫 선생님이 기존 강의실을 claim하는 마이그레이션 규칙이 필요하다.
- 기존 세션은 강의별 1개였으므로 학생별 세션 분리를 구현하지 않으면 학습 기록이 섞인다.
- `/uploads`가 공개 정적 경로로 남아 있으면 학생이 직접 URL로 자료를 볼 수 있다.
- Google OAuth는 실제 자격증명이 없으면 브라우저에서 최종 성공까지 테스트할 수 없다.
