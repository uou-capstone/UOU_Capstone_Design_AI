# 0426 업데이트 계획서: 로그인, 회원가입, 역할별 강의실 권한

## 목표

이 업데이트의 목표는 현재 단일 사용자처럼 동작하는 강의실 시스템을 선생님과 학생이 함께 쓰는 계정 기반 서비스로 확장하는 것이다. 선생님은 지금처럼 강의실과 자료를 만들 수 있고, 학생은 초대받은 강의실에서 선생님이 올린 자료만 학습할 수 있어야 한다.

## 핵심 원칙

- [ ] 인증은 가능하면 검증된 무료 또는 저비용 인증 서비스를 우선 검토한다.
- [ ] 무료 정책, 사용량 제한, 이메일 인증 정책은 구현 시작 전에 공식 문서에서 다시 확인한다.
- [ ] 로그인 여부와 역할 권한은 프론트엔드 UI가 아니라 서버 API에서 최종 검증한다.
- [ ] 학생 초대용 `이름 #1234` 코드는 사용자를 찾기 쉽게 하는 식별 보조값으로만 사용한다.
- [ ] 실제 강의실 접근 권한은 사용자 ID, 역할, 초대 또는 등록 기록을 기준으로 판정한다.
- [ ] 학생 개인정보와 학습 기록은 필요한 범위만 저장하고, 로그에 민감정보가 남지 않도록 한다.

## 1. 로그인 기능 구현

### 인증 방식 선정

- [ ] 후보 인증 방식을 비교한다.
  - [ ] Firebase Authentication
  - [ ] Supabase Auth
  - [ ] Auth.js 기반 자체 서버 인증
  - [ ] 프로젝트 유지비, 무료 사용 가능 범위, 이메일 인증, Google 로그인, 로컬 개발 편의성을 비교한다.
- [ ] 기본 권장안 하나를 정한다.
  - [ ] 외부 인증 서비스를 쓰는 경우: 비밀번호 저장, 이메일 인증, Google OAuth는 인증 서비스에 위임한다.
  - [ ] 자체 인증을 쓰는 경우: 비밀번호 해시, 세션 저장소, 이메일 발송, 토큰 만료, 재설정 기능까지 직접 구현한다.
- [ ] `.env`에 인증 관련 값을 분리한다.
  - [ ] 클라이언트 공개 키와 서버 전용 비밀 키를 분리한다.
  - [ ] 서버 전용 키는 프론트엔드 번들에 절대 포함하지 않는다.

### 로그인 UX

- [ ] 로그인 화면을 추가한다.
- [ ] 이메일 로그인 폼을 제공한다.
- [ ] Google 계정 로그인 버튼을 제공한다.
- [ ] 로그인 실패 원인을 사용자가 이해할 수 있게 표시한다.
- [ ] 로그인 중 로딩 상태를 명확하게 보여준다.
- [ ] 로그인한 사용자는 대시보드로 이동한다.
- [ ] 로그인하지 않은 사용자가 강의실이나 세션 URL에 접근하면 로그인 화면으로 이동한다.
- [ ] 로그아웃 버튼을 제공한다.

### 서버 보안

- [ ] 모든 보호 API에서 인증 토큰 또는 세션을 검증한다.
- [ ] 인증되지 않은 요청은 `401 Unauthorized`로 거부한다.
- [ ] 권한이 부족한 요청은 `403 Forbidden`으로 거부한다.
- [ ] 토큰 만료, 변조, 누락 케이스를 테스트한다.
- [ ] CORS 설정을 실제 프론트엔드 주소 기준으로 제한한다.
- [ ] 로그인, 회원가입, 초대 관련 API에 rate limit을 적용한다.
- [ ] 서버 로그에 토큰, 비밀번호, 이메일 인증 링크가 남지 않도록 한다.

## 2. 회원가입 구현

### 가입 방식

- [ ] 이메일 회원가입을 제공한다.
- [ ] 이메일 인증이 완료된 계정만 실제 서비스 접근을 허용한다.
- [ ] Google 계정으로 회원가입할 수 있게 한다.
- [ ] 동일 이메일이 중복 가입되지 않도록 한다.
- [ ] 회원가입 직후 기본 프로필을 생성한다.

### 필수 프로필

- [ ] 사용자 ID를 저장한다.
- [ ] 이메일을 저장한다.
- [ ] 표시 이름을 저장한다.
- [ ] 역할을 저장한다: `teacher` 또는 `student`.
- [ ] 학생 초대 검색용 4자리 코드를 생성한다.
- [ ] 생성 시각과 마지막 수정 시각을 저장한다.

### 4자리 코드 정책

- [ ] 학생 계정마다 화면에 `이름 #1234` 형태의 초대 태그를 보여준다.
- [ ] 4자리 코드는 숫자 또는 대문자 혼합 중 하나로 정책을 정한다.
- [ ] 같은 이름과 같은 코드 조합이 중복되지 않도록 한다.
- [ ] 코드 재생성 기능이 필요한지 결정한다.
- [ ] 4자리 코드는 짧으므로 비밀 초대 토큰처럼 취급하지 않는다.
- [ ] 학생 초대 확정은 서버가 학생의 실제 사용자 ID를 찾아 등록하는 방식으로 처리한다.

## 3. 역할 선택 회원가입

### 가입 단계

- [ ] 회원가입 중 역할 선택 단계를 추가한다.
- [ ] 선생님 계정과 학생 계정을 명확히 구분해서 설명한다.
- [ ] 역할 선택 후 되돌아가 수정할 수 있게 한다.
- [ ] 가입 완료 후 역할별 첫 화면으로 이동한다.

### 역할 변경 정책

- [ ] 사용자가 임의로 학생에서 선생님으로 바꿀 수 있는지 정책을 정한다.
- [ ] 기본 권장안: 가입 후 일반 사용자는 역할을 직접 변경할 수 없게 한다.
- [ ] 역할 변경이 필요하면 관리자 승인 또는 별도 관리 기능으로 처리한다.

## 4. 선생님 기능

### 강의실 관리

- [ ] 선생님은 강의실을 만들 수 있다.
- [ ] 선생님은 본인이 만든 강의실만 볼 수 있다.
- [ ] 선생님은 본인이 만든 강의실만 수정하거나 삭제할 수 있다.
- [ ] 선생님은 본인 강의실에 주차를 만들 수 있다.
- [ ] 선생님은 본인 강의실의 주차에 PDF 자료를 추가할 수 있다.
- [ ] 선생님은 본인이 올린 자료를 삭제할 수 있다.

### 학생 초대

- [ ] 강의실 상세 화면에 학생 초대 영역을 추가한다.
- [ ] 선생님은 학생의 `이름`과 `#4자리 코드`를 입력해 학생을 찾을 수 있다.
- [ ] 검색 결과에는 이름, 코드, 이메일 일부 마스킹 값 등 확인 가능한 최소 정보만 보여준다.
- [ ] 선생님이 초대 확정 버튼을 누르면 해당 학생을 강의실에 등록한다.
- [ ] 이미 등록된 학생을 다시 초대하면 중복 등록하지 않고 안내 메시지를 보여준다.
- [ ] 없는 학생 또는 코드 불일치 시 명확한 오류를 보여준다.
- [ ] 초대 목록에서 학생을 제거할 수 있는지 정책을 정한다.
- [ ] 기본 권장안: 선생님은 본인 강의실에서 학생을 제거할 수 있다.

### 학생 목록

- [ ] 강의실별 학생 목록을 보여준다.
- [ ] 학생 이름과 초대 코드, 등록일을 보여준다.
- [ ] 학생별 학습 리포트 또는 진행률과 연결할 수 있는 자리를 마련한다.
- [ ] 학생 개인정보는 필요한 범위만 노출한다.

## 5. 학생 기능과 제한

### 학생 대시보드

- [ ] 학생은 초대받은 강의실만 볼 수 있다.
- [ ] 초대받지 않은 강의실은 URL을 직접 입력해도 접근할 수 없다.
- [ ] 학생은 강의실 생성 버튼을 볼 수 없다.
- [ ] 학생은 주차 생성, 자료 업로드, 삭제 버튼을 볼 수 없다.
- [ ] 학생은 선생님이 올린 자료만 열람할 수 있다.
- [ ] 학생은 본인이 학습한 세션 기록과 퀴즈 기록을 이어서 볼 수 있다.

### 서버 권한 제한

- [ ] 학생의 `POST /classrooms` 요청을 서버에서 차단한다.
- [ ] 학생의 `POST /weeks` 요청을 서버에서 차단한다.
- [ ] 학생의 `POST /lectures` 업로드 요청을 서버에서 차단한다.
- [ ] 학생의 삭제 요청을 서버에서 차단한다.
- [ ] 학생은 등록된 강의실의 자료 조회와 학습 세션 API만 사용할 수 있다.
- [ ] 학생은 다른 학생의 학습 기록을 조회할 수 없다.

## 6. 데이터 모델 변경

### 새로 필요한 데이터

  - [ ] `id`
  - [ ] `email`
  - [ ] `displayName`
  - [ ] `role`
  - [ ] `inviteCode`
  - [ ] `createdAt`
  - [ ] `updatedAt`
- [ ] `classrooms`에 소유자 정보를 추가한다.
  - [ ] `teacherId`
- [ ] 강의실 학생 등록 데이터를 추가한다.
  - [ ] `classroomId`
  - [ ] `studentId`
  - [ ] `invitedByTeacherId`
  - [ ] `createdAt`
- [ ] 기존 강의실 데이터 마이그레이션 방식을 정한다.
  - [ ] 개발용 기존 데이터는 기본 선생님 계정에 연결한다.
  - [ ] 마이그레이션 실패 시 백업을 남긴다.

### 저장소 선택

- [ ] 현재 JSON 파일 저장 방식을 유지할지, DB로 옮길지 결정한다.
- [ ] 인증 서비스가 DB를 제공하면 사용자 프로필과 등록 관계를 그 DB에 둘지 검토한다.
- [ ] JSON 저장을 유지하는 경우 동시 쓰기, 백업, 마이그레이션 테스트를 강화한다.
- [ ] 운영 사용을 염두에 두면 Postgres 계열 DB를 우선 검토한다.

## 7. API 변경 계획

### 인증 API

- [ ] 현재 사용자 조회 API를 추가한다.
  - [ ] `GET /api/me`
- [ ] 프로필 생성 또는 동기화 API를 추가한다.
  - [ ] `POST /api/me/profile`
- [ ] 서버 공통 인증 미들웨어를 추가한다.
- [ ] 서버 공통 역할 검사 미들웨어를 추가한다.

### 강의실 API

- [ ] `GET /api/classrooms`가 역할별로 다른 결과를 반환하게 한다.
  - [ ] 선생님: 본인이 만든 강의실
  - [ ] 학생: 등록된 강의실
- [ ] `POST /api/classrooms`는 선생님만 허용한다.
- [ ] `DELETE /api/classrooms/:id`는 해당 강의실 소유 선생님만 허용한다.
- [ ] 주차와 강의 자료 API에도 같은 권한 검사를 적용한다.

### 초대 API

- [ ] 학생 검색 API를 추가한다.
  - [ ] `GET /api/students/search?name=...&code=...`
- [ ] 강의실 학생 등록 API를 추가한다.
  - [ ] `POST /api/classrooms/:classroomId/students`
- [ ] 강의실 학생 목록 API를 추가한다.
  - [ ] `GET /api/classrooms/:classroomId/students`
- [ ] 강의실 학생 제거 API를 추가할지 결정한다.
  - [ ] `DELETE /api/classrooms/:classroomId/students/:studentId`

## 8. UI 및 UX 개선

### 공통 UX

- [ ] 앱 진입 시 로그인 상태를 자연스럽게 확인한다.
- [ ] 인증 확인 중에는 빈 화면 대신 로딩 화면을 보여준다.
- [ ] 역할별로 불필요한 버튼과 설명을 숨긴다.
- [ ] 에러 메시지는 사용자가 다음 행동을 알 수 있게 작성한다.
- [ ] 빈 상태 화면을 개선한다.
  - [ ] 선생님: 첫 강의실 만들기 안내
  - [ ] 학생: 아직 초대받은 강의실이 없다는 안내
- [ ] 모바일 화면에서 대시보드, 강의실, 세션이 깨지지 않게 한다.
- [ ] 키보드 접근성과 포커스 이동을 확인한다.
- [ ] 색상 대비와 버튼 크기를 점검한다.

### 선생님 UX

- [ ] 대시보드에 “내 강의실” 흐름을 명확히 만든다.
- [ ] 강의실 생성, 주차 생성, 자료 업로드가 한눈에 이어지게 한다.
- [ ] 학생 초대 입력 폼은 `이름`과 `#코드`를 분리해서 받는다.
- [ ] 초대 성공 후 학생 목록이 즉시 갱신되게 한다.
- [ ] 초대 실패 시 어떤 값이 문제인지 알려준다.

### 학생 UX

- [ ] 학생 첫 화면에는 초대받은 강의실만 노출한다.
- [ ] 학생이 자료를 열면 기존 학습 세션으로 자연스럽게 이어진다.
- [ ] 학생에게는 수정, 삭제, 업로드 액션이 보이지 않게 한다.
- [ ] 초대 코드 확인 위치를 프로필 또는 계정 메뉴에 제공한다.

## 9. 테스트 체크리스트

### 인증 테스트

- [ ] 이메일 회원가입이 된다.
- [ ] 이메일 인증 전에는 보호 화면에 접근할 수 없다.
- [ ] 이메일 인증 후 로그인할 수 있다.
- [ ] Google 계정으로 가입하고 로그인할 수 있다.
- [ ] 로그아웃 후 보호 URL 접근 시 로그인 화면으로 이동한다.
- [ ] 만료되거나 잘못된 토큰은 서버에서 거부된다.

### 역할 테스트

- [ ] 선생님 계정은 강의실을 만들 수 있다.
- [ ] 학생 계정은 강의실 만들기 버튼이 보이지 않는다.
- [ ] 학생 계정이 직접 API로 강의실 생성을 요청해도 거부된다.
- [ ] 선생님은 본인 강의실만 볼 수 있다.
- [ ] 학생은 초대받은 강의실만 볼 수 있다.
- [ ] 학생은 초대받지 않은 강의실 URL 접근이 거부된다.

### 초대 테스트

- [ ] 학생 가입 후 `이름 #1234` 초대 태그가 생성된다.
- [ ] 선생님이 올바른 이름과 코드로 학생을 초대할 수 있다.
- [ ] 잘못된 코드로 초대하면 실패한다.
- [ ] 같은 학생을 중복 초대하면 중복 등록되지 않는다.
- [ ] 초대된 학생은 해당 강의실을 볼 수 있다.
- [ ] 초대되지 않은 학생은 해당 강의실을 볼 수 없다.

### 자료 접근 테스트

- [ ] 선생님은 PDF를 업로드할 수 있다.
- [ ] 학생은 PDF를 업로드할 수 없다.
- [ ] 학생은 초대받은 강의실의 PDF를 열람할 수 있다.
- [ ] 학생은 다른 강의실의 PDF URL을 직접 입력해도 열람할 수 없다.
- [ ] 학생별 학습 세션과 퀴즈 기록이 분리된다.

### 회귀 테스트

- [ ] 기존 강의실 생성 흐름이 선생님 계정에서 정상 동작한다.
- [ ] 기존 주차 생성 흐름이 선생님 계정에서 정상 동작한다.
- [ ] 기존 PDF 업로드 흐름이 선생님 계정에서 정상 동작한다.
- [ ] 기존 학습 세션, 페이지 이동, 퀴즈, 저장 후 재입장이 정상 동작한다.
- [ ] 서버 테스트와 웹 빌드가 통과한다.

## 10. 구현 순서 제안

- [ ] 1단계: 인증 제공 방식 결정과 환경 변수 설계
- [ ] 2단계: 사용자 프로필, 역할, 강의실 소유자, 학생 등록 데이터 모델 추가
- [ ] 3단계: 서버 인증 미들웨어와 역할 권한 미들웨어 구현
- [ ] 4단계: 강의실, 주차, 자료, 세션 API에 권한 검사 적용
- [ ] 5단계: 로그인, 회원가입, 역할 선택 화면 구현
- [ ] 6단계: 선생님 대시보드와 학생 대시보드 분기
- [ ] 7단계: 학생 초대 검색, 등록, 목록 UI 구현
- [ ] 8단계: 기존 데이터 마이그레이션과 회귀 테스트
- [ ] 9단계: 실제 브라우저에서 선생님과 학생 계정을 모두 사용해 E2E 점검
- [ ] 10단계: UX 다듬기와 오류 메시지 정리

## 완료 기준

- [ ] 사용자는 이메일 또는 Google 계정으로 가입하고 로그인할 수 있다.
- [ ] 회원가입 시 선생님 또는 학생 역할을 선택할 수 있다.
- [ ] 선생님은 본인 강의실을 만들고 자료를 올릴 수 있다.
- [ ] 선생님은 학생의 `이름 #4자리 코드`로 강의실에 학생을 초대할 수 있다.
- [ ] 학생은 초대받은 강의실만 볼 수 있다.
- [ ] 학생은 강의실 생성, 주차 생성, 자료 업로드, 삭제를 할 수 없다.
- [ ] 모든 권한 제한은 프론트엔드 숨김 처리와 서버 검증을 모두 통과한다.
- [ ] 기존 교육 에이전트 학습 흐름은 선생님과 학생 권한 체계 안에서도 정상 동작한다.
