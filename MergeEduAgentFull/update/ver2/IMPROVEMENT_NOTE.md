# Ver2 개선점 메모

- 작성일: `2026-04-15`
- 대상: `Quiz Diagnosis Ver2`
- 성격: `현재 버전은 동작하지만, 다음 개선 우선순위를 정리한 메모`

## 현재 판단

Ver2는 현재 기준으로 `동작 가능`하고,
테스트/빌드/정적 리뷰도 통과했다.

하지만 아래 항목들은 다음 단계에서 개선하면 더 좋아진다.

추가로,
이번 멀티 에이전트 검증에서는
`기능 흐름 자체는 대체로 PASS`
였지만,
`테스트 방어막은 아직 보강 필요`
라는 의견이 함께 나왔다.

즉,
`실제로 동작한다`와
`앞으로도 회귀 없이 유지된다`
를 분리해서 볼 필요가 있다.

## 1. 진단 정확도 개선

현재 진단은 주로 아래 신호를 사용한다.

- 현재 오답 문항
- 점수 비율
- 같은 페이지의 최근 저득점 이력
- 기존 약점 메모

이 구조는 빠르고 가볍다는 장점이 있지만,
아직 `정확한 개념 태그 기반 진단`은 아니다.

개선 방향:

- 퀴즈 문항에 concept tag를 넣기
- 채점 결과에도 concept 단위 근거를 남기기
- 페이지 기준이 아니라 concept 기준 반복 오답을 추적하기

## 2. 반복 오답 추적 고도화

현재는 같은 페이지의 최근 저득점 이력을 보는 수준이다.

즉,
`같은 페이지에서 또 흔들렸다`
는 건 볼 수 있지만,
`다른 페이지지만 사실 같은 개념을 계속 틀린다`
는 건 아직 약하다.

개선 방향:

- concept 단위 반복 오답 카운트 추가
- 페이지가 달라도 같은 약점으로 묶는 규칙 추가
- 최근성 가중치 추가

## 3. 메모리 품질 개선

현재 메모리 write는 의도적으로 짧고 보수적이다.
이건 안전하지만, 표현력이 아직 제한적이다.

현재 한계:

- confidence가 약함
- provenance가 없음
- decay가 없음
- `왜 그렇게 저장됐는지` 근거가 희미함

개선 방향:

- 메모리 항목별 confidence 추가
- 어떤 퀴즈/질문에서 나온 근거인지 기록
- 오래된 약점은 약해지는 decay 규칙 추가
- 같은 개념을 이후 재시험이나 QA에서 잘 풀면 약점을 완화하거나 정리하는 규칙 추가
- `약점이 계속 남기만 하는 구조`에서 `개선되면 반영되는 구조`로 확장

## 4. 교정 설명 전용성 강화

현재 `MisconceptionRepairAgent`는
기존 QA 브리지를 감싸는 서버 래퍼 방식이다.

장점:

- 빠르게 적용 가능
- 새 Python 엔드포인트 없이 검증 쉬움

한계:

- 교정 전용 프롬프트/평가 체계가 아직 완전히 독립적이지 않음

개선 방향:

- 전용 repair prompt 더 세분화
- 교정 설명 품질 rubric 추가
- 필요 시 전용 bridge endpoint 분리 검토

## 5. UI 가시성 개선

현재는 새 위젯 없이 기존 채팅 입력으로 동작한다.
실용적이지만, 사용자 입장에서는
`지금 내가 일반 질문을 하는 중인지, 진단 답변을 하는 중인지`
헷갈릴 수 있다.

개선 방향:

- 진단 대기 상태 배지 표시
- `지금은 진단 답변 단계입니다` 안내 문구 강화
- 교정 후 요약 카드 또는 상태 표시 추가
- 진단 대기 중 일반 질문을 교정 답변으로 오인하지 않도록 짧은 의도 확인 추가
- `잘 모르겠어요`, `넘어갈래요` 같은 응답에 대한 분기 문구 보강

## 6. 운영 지표 추가

현재는 기능은 동작하지만,
이 변화가 실제로 교육적으로 더 좋은지 볼 운영 지표가 아직 부족하다.

개선 방향:

- diagnosis 진입률
- diagnosis 후 재시험 성공률
- 같은 개념 재오답 비율
- overhelping 비율
- 교정 후 다음 질문 감소/증가 패턴

## 7. QA/설명 개인화 활용도 강화

현재 교정 후 메모리는 이후 설명/QA/퀴즈에 전달된다.
다만 설명/퀴즈보다 QA 쪽은
`이 메모리를 어떻게 활용해야 하는지`
프롬프트 규칙이 상대적으로 덜 선명하다.

개선 방향:

- QA 프롬프트에 `최근 약점이면 바로 정답만 주지 말고 한 번 더 짚기` 같은 규칙 추가
- 설명 프롬프트에도 `next coaching goals`를 더 직접적으로 사용하도록 명시
- 교정 직후 1~2턴 동안은 최근 약점을 우선 반영하는 recency 규칙 추가

## 8. API / 세션 동기화 단순화

현재 퀴즈 제출 뒤 흐름은
`이벤트 처리 -> 부분 patch 반영 -> 전체 세션 재조회`
조합에 기대는 부분이 있다.

일반적인 상황에서는 잘 동작하지만,
프론트엔드 입장에서는
`채점 결과`, `activeIntervention`, `퀴즈 기록`이
한 번에 완전히 동기화되지 않는 구조라
구현이 다소 복잡해진다.

개선 방향:

- 이벤트 응답에 갱신된 `quiz record`를 직접 포함
- 또는 `quizPatch` / `gradedQuiz` 같은 전용 patch 추가
- 퀴즈 모달이 전체 세션 refetch 없이도 완결되게 만들기

## 9. 테스트 범위 확장

현재는 서버 테스트와 빌드가 통과했다.
이건 충분히 좋은 출발점이다.

하지만 다음 단계에서는 아래가 있으면 더 좋다.

- 오케스트레이션 엔진 레벨 통합 테스트
  - `저득점 -> activeIntervention 생성 -> patch 반영 -> 다음 USER_MESSAGE -> repair 실행`
- StateReducer cleanup 테스트 확장
  - `다음 페이지`, `설명 재시작`, `복습 결정`, `재시험 결정`
- 서술형 채점 경로 테스트
  - `GRADE_SHORT_OR_ESSAY` 저득점 진단 진입
  - 합격 시 intervention 해제
- 최소 프런트 통합 테스트
  - `response.patch.activeIntervention`가 세션 상태에 제대로 merge되는지
- 실제 legacy session JSON migration 샘플 테스트
  - `activeIntervention` 필드가 아예 없는 예전 세션 파일 로드
- 스트리밍 중 진단 상태 변경 테스트
- 장기 세션에서 메모리 누적 품질 테스트

중요:

현재 리뷰 기준으로는
`구현은 PASS에 가깝지만, 테스트 방어막은 FAIL 판정이 나올 수 있는 상태`
였다.

그래서 다음 단계의 우선순위에는
테스트 확장이 반드시 포함되어야 한다.

## 10. 상태 계약과 수명주기 보강

이번 리뷰에서는
`activeIntervention` 상태 계약을 더 단단하게 하면 좋겠다는 의견도 있었다.

개선 방향:

- API patch의 `activeIntervention`을 optional보다 더 명시적인 계약으로 고정 검토
- 새 퀴즈를 시작할 때 이전 `REPAIR_DELIVERED` 상태를 정리할지 검토
- legacy session 로드시 malformed intervention object sanitize 검토

## 11. 추천 우선순위

가장 먼저 할 것:

1. 멀티턴 handoff 통합 테스트와 cleanup 테스트 보강
2. concept tag 기반 진단 보강
3. 반복 오답 추적을 page 기준에서 concept 기준으로 확장
4. 약점 decay / 회복 반영 규칙 추가
5. 진단/교정 상태의 UI 가시성 및 의도 확인 강화
6. 퀴즈 제출 이후 API/session sync 단순화

그 다음 할 것:

7. 메모리 confidence / provenance 추가
8. QA/설명 개인화 규칙 강화
9. 상태 계약과 intervention 수명주기 보강
10. 운영 지표와 evaluator 추가
11. 필요 시 repair 전용 bridge 분리

네 가지다.
