## 테스트 목적

- 선생님/학생 역할 분리가 실제 사용자 흐름에서 유지되는지 확인한다.
- 강의실, 주차, 강의 자료, 학생 초대, 학습 세션, 리포트 기능이 의도한 권한 경계 안에서 동작하는지 확인한다.
- Comet 브라우저를 마우스/키보드/스크롤로 직접 조작하면서 사용자가 느낄 UI/UX 불편사항을 함께 기록한다.
- 이번 문서는 수정 지시서가 아니라, 어떤 시나리오가 통과/실패했는지와 실패 이유를 남기는 테스트 로그다.

## 테스트 계정

- 교사 A: `safety.teacher.a.0427@example.com`
- 교사 B: `safety.teacher.b.0427@example.com`
- 학생 A: `safety.student.a.0427@example.com`
- 학생 B: `safety.student.b.0427@example.com`
- 학생 C: `safety.student.c.0427@example.com`

비밀번호는 모두 로컬 테스트용 동일 비밀번호를 사용한다.

## 실행 상태 표기

- `PENDING`: 아직 실행 전
- `PASS`: 기대 결과와 일치
- `FAIL`: 기대 결과와 다름
- `PARTIAL`: 핵심 권한/상태는 맞으나 일부 UI/UX 또는 보조 검증 미흡
- `BLOCKED`: 외부 환경이나 도구 한계로 실행 불가

## UI/UX 관찰 로그

- 학생 초대 목록과 리포트 카드에서 긴 학생 이름, 초대 코드, 마스킹 이메일이 한 줄/한 카드에 촘촘하게 배치되어 가독성이 떨어진다.
- 리포트 지표 카드의 `완료 페이지 1/71p`와 리포트 본문/세션 데이터의 `2/71페이지`가 서로 달라 교사가 진도 판단을 할 때 혼동될 수 있다.
- 퀴즈 모달은 정상 동작하지만 세로로 길어져 스크롤 위치가 내려가면 현재 몇 번째 문항을 보고 있는지 맥락이 약해진다. 상단 고정 진행 표시가 있으면 좋다.
- 학습 세션 첫 안내 메시지가 `현재 1페이지입니다`라고 남은 뒤, 바로 다음 페이지로 이동하면 이후 페이지 이동 메시지가 추가로 붙는다. 기능은 맞지만 이전 안내가 눈에 남아 학생이 헷갈릴 수 있다.
- 학생 대시보드와 강의실 화면은 권한 분리가 잘 되지만, 접근 권한 오류 화면에는 홈/대시보드로 돌아가는 주요 행동 버튼이 더 눈에 띄게 있으면 좋겠다.
- 채팅 라우터가 문장 안의 `다음 페이지` 표현만 보고 실제 명령이 아닌 설명 문맥까지 페이지 이동으로 분류한다. 학생이 "다음 페이지를 누르면 어떻게 돼?"처럼 말해도 의도치 않게 페이지가 넘어갈 수 있어 확인/분류 안정성이 필요하다.
- 단답형 퀴즈 모달이 새로 열릴 때 빈 답안인데도 즉시 `총점 100/100`과 `채점 완료`가 표시되는 상태가 관찰되었다. 학생 입장에서는 무엇을 입력해야 하는지보다 이미 끝난 시험처럼 보이며, 교사용 리포트의 퀴즈 성과 데이터도 왜곡될 수 있다.
