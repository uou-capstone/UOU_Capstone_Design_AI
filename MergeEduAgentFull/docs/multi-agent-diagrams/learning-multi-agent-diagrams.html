학습 멀티 에이전트 전체 컨텍스트

현재 구조는 LLM들이 서로 임의로 대화하는 모델이 아니라, 오케스트레이터가 제한된 tool call 계획을 만들고 서버가 검증한 뒤 전문 에이전트를 순차 실행하는 구조다.

기준 코드: OrchestrationEngine, Orchestrator, ToolDispatcher, QuizAssessmentService, QuizDiagnosisService, GeminiBridgeClient
1. Event

Session UI가 event/stream으로 사용자 행동을 보낸다.

2. Reduce

StateReducer가 페이지, 메시지, 상태를 먼저 반영한다.

3. Context

PDF 페이지 텍스트, 메모리, QA thread, assessment를 압축한다.

4. Plan

Orchestrator가 pedagogyPolicy와 tool actions JSON을 만든다.

5. Verify

ToolDispatcher가 policy, budget, hard cap을 적용한다.

6. Agents

설명, QA, 퀴즈, 채점, 오답교정 에이전트를 호출한다.

7. Persist

SessionState, summary, assessment, quiz log를 저장한다.

시나리오 1. 세션 진입과 페이지 설명

처음 들어오면 바로 설명하지 않고 시작 여부를 묻고, 동의 후에는 현재 페이지 설명과 다음 학습 선택지를 만든다.

주요 코드: StateReducer, Orchestrator.fallback, EXPLAIN_PAGE, ExplainerAgent

  • START_EXPLANATION_DECISION accept=true면 EXPLAIN_PAGE가 실행된다.
  • 설명 뒤에는 퀴즈 또는 다음 페이지 결정을 유도한다.

시나리오 2. 자유 질문과 QA follow-up

학생 질문은 일반 설명 재시작이 아니라 ANSWER_QUESTION으로 라우팅되고, 같은 페이지 후속 질문만 짧은 thread로 이어진다.

주요 코드: QaThreadService, ANSWER_QUESTION, QaAgent

  • 새 질문이면 START_NEW로 thread를 비운다.
  • QA에는 설명 전문 전체가 아니라 이전 QA 문맥 최대 6턴만 전달된다.

시나리오 4. 채점, assessment, 다음 턴 handoff

채점 결과는 quiz.grading에 저장되고, 별도의 quizAssessment가 PENDING 상태로 만들어져 다음 턴 planner 입력으로 전달된다.

주요 코드: AUTO_GRADE_MCQ_OX, GRADE_SHORT_OR_ESSAY, QuizAssessmentService

두 채점 경로

  • MCQ/OX는 pending assessment가 없으면 LLM 없이 deterministic fast path.
  • SHORT/ESSAY는 GraderAgent가 Gemini로 채점한다.

시나리오 6. 안전장치와 fallback

모델 호출이 실패하거나 JSON plan이 깨져도 deterministic fallback과 verifier가 같은 계약을 유지한다.

주요 코드: planWithLlm catch, normalizePlan, verifyAndPatchPlan, AbortSignal

fallback 조건

  • SAVE_AND_EXIT.
  • geminiFile 없음.
  • bridge/model 실패 또는 plan parsing 실패.
  • pending assessment 없는 MCQ/OX 제출 fast path.
